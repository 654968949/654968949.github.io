企业级RAG系统搭建实战：基于Spring AI的多路召回方案

前言：为什么需要企业级RAG？

通用大模型虽然能力很强，但它不知道你们公司内部的接口规范、运维手册、产品文档。直接把内部文档丢给大模型做微调？成本高、周期长、更新也麻烦。RAG（Retrieval-Augmented Generation）是目前更务实的选择——检索 + 生成的组合，让大模型在回答之前先从知识库里”翻一翻”相关资料，再做回答。

听起来简单，但真正搭一套能跑在企业环境里的 RAG 系统，要解决的事不少：

  • 单路向量召回覆盖面不够，关键词匹配同样重要
  • 文档要支持 PDF、Word 等多种格式的摄入
  • 多路结果怎么融合排序才能最大化相关度
  • 高并发场景下检索延迟要可控
  • 敏感词过滤、权限管控这些工程化问题一个不能少

这篇文章基于我实际搭建的 woods-spring-ai-rag 项目，完整走一遍企业级 RAG 系统的搭建过程。

一、整体架构

先看全局。系统分为三大层：

文档摄入层 → 文件上传 → 文本提取 → 分片 → 分别写入 MySQL、Milvus 向量库、Elasticsearch 关键词索引。

检索召回层 → 用户提问进来，并行走三条通道：向量语义召回（Milvus）、关键词召回（ES）、数据库包含式召回（MySQL LIKE/FULLTEXT），然后通过 RRF（Reciprocal Rank Fusion）融合排序，可选叠加 Reranker 重排和 MMR 去冗余。

问答服务层 → 融合后的知识片段拼成上下文 Prompt，调用 LLM 生成回答，支持同步和流式两种模式。

技术栈方面：Spring Boot 3.5.13 + Spring AI 1.1.4，LLM 用的是阿里 DashScope 的 qwen3.6-plus，Embedding 模型是 text-embedding-v4（1536 维），向量库选 Milvus，关键词检索用 ES 9.3.0。

二、文档摄入管道

企业文档格式五花八门，第一步是统一提取文本。

2.1 多格式文本提取

通过 FileTextExtractorService 统一入口，内部根据文件类型路由到不同解析器：

  • TXT → 直接按 UTF-8 读取
  • PDF → Apache PDFBox 逐页提取
  • DOCX → Apache POI XWPF 解析段落和表格
  • 通用格式 → Apache Tika 兜底自动检测

所有提取器在 FileTextExtractorServiceImpl 中注入为 List<FileTextExtractor>，按 canHandle() 匹配，扩展新格式只需新增一个实现类。

2.2 文本分片策略

提取出全文后，不能直接整篇塞给 LLM——Token 有限，而且太长的上下文会稀释关键信息。分片逻辑在 KnowledgeTextSplitter 中：

分片时会优先在句号、换行等自然断点切割，避免把一句话拦腰截断。重叠 120 字符确保边界附近的上下文不会丢失。分片后同时写三份：

存储 用途
MySQL (knowledge_document_chunk) 元数据管理、数据库级召回
Milvus (rag_knowledge) 向量语义检索（1536 维 COSINE）
Elasticsearch (rag-knowledge-chunks) 关键词全文检索（ngram 2-3 分词）

Embedding 调用做了批量优化，每批 10 条发送给 DashScope，避免单条调用的网络开销。

三、多路召回策略（核心）

这是整个系统最关键的部分。单靠向量召回在某些场景下效果不好——比如用户搜”数据库连接池配置”，向量模型可能给它召回一堆”数据库优化”的泛内容，反而漏掉了标题精确匹配的那篇文档。

我们的方案是三路并行召回 + RRF 融合。

3.1 三路召回通道

每条通道可以独立开关、独立配置 Top-K 和融合权重：

三路各司其职：

  • 向量召回（权重 1.0）：语义相似度，擅长理解”意思相近但措辞不同”的内容
  • 关键词召回（权重 0.8）：ES ngram 分词 + 多字段匹配，擅长精确术语命中
  • 数据库召回（权重 0.6）：MySQL LIKE/FULLTEXT 兜底，不依赖外部中间件也能跑

向量权重 > 关键词 > 数据库，因为语义理解通常最可靠，但不完全依赖某一路。

四、RRF 融合与检索后优化

4.1 RRF 融合排序

三路召回的结果如何合并成一份排序列表？我们用的是 RRF（Reciprocal Rank Fusion），支持两种模式：

WEIGHTED 模式（默认）：

STANDARD 模式：

k=60 是经典平滑常数，防止 rank=1 的结果主导整个列表。加权模式下向量通道的 rank=1 得分为 1.0/(60+1)=0.0164，关键词通道 rank=1 为 0.8/61=0.0131，差序合理。

同一分片被多路同时命中时，分数累加，多路命中天然排序靠前——这个设计很符合直觉，”大家都说好的内容更靠谱”。

4.2 Reranker 重排序

RRF 是无监督公式，有时候排序不够精准。系统提供了两种可选的 Reranker：

  • SimpleReranker（默认，@Primary）：基于 2-gram Jaccard 字符相似度 + RRF 分数 50/50 混合，纯算法无外部依赖
  • LlmPointwiseReranker：让 LLM 对每个候选分片打 0-100 的相关性分，通过专用线程池并行评估

Reranker 默认关闭，适合在候选数量较多（>topK）时开启，用精排提升最终结果质量。

4.3 MMR 多样性优化

有时候一个文档的多个相邻分片都排进了 Top-K，但它们的内容高度重复。MMR（Maximal Marginal Relevance）通过贪心算法做去冗余：

同一文档的不同分片视为完全冗余（相似度 1.0），lambda=0.7 表示偏向相关性但兼顾多样性。这样可以避免 LLM 看到一堆重复上下文。

4.4 查询改写（可选）

用户提问有时候太口语化，比如”上次那个数据库挂了怎么搞的”——这不利于检索。启用查询改写后，LLM 会生成最多 2 个语义变体：

每个变体独立执行一遍多路召回，结果跨变体去重合并，提升召回覆盖率。这个功能默认关闭，因为多了 LLM 调用会增加 1-2 秒延迟，按需开启。

每次问答前，SensitiveWordService 扫描用户问题：

被拦截的问题也会持久化到 qa_record，标记 is_blocked=1，方便审计。

5.3 鉴权与 RBAC

基于 Bearer Token 的认证机制，AuthInterceptor 拦截所有 /api/** 请求。首次启动时 AuthBootstrapRunner 自动创建管理员账号（默认 admin / Admin@123456，168 小时会话有效期）。

管理类接口（用户管理、敏感词管理等）通过 @RequiresAdmin 注解 + AdminAuthorizationInterceptor 做角色校验。

5.4 优雅降级

Milvus 不可用时，应用不会启动失败——MilvusConfiguration 返回一个 UnavailableVectorStore 占位，调用时抛出 BusinessException(503)，其他通道继续工作。同样，ES 连接不上时关键词召回静默降级，不影响整体服务可用性。

这种设计在中小团队很有意义——不是所有环境都跑得动完整的 Milvus + ES 集群，搭什么用什么，不搭的自动跳过。

5.5 离线评估

系统内置了一套检索质量评估框架，加载 JSON 格式的评测数据集，自动计算每道题的 Recall@5、MRR、NDCG@5：

评估接口 POST /api/rag-evaluation/evaluate 一键跑完整个数据集，输出聚合指标。调参的时候跑一遍，立刻知道改了什么效果。

小结

这套方案的几个关键取舍：

  1. 多路召回 > 单路召回：向量 + 关键词 + 数据库的组合，比单一向量检索覆盖面广得多
  2. RRF 加权融合：简单有效，没有训练成本，权重可调可观测
  3. 全链路可配置：每个通道、每个优化环节都可以通过 YAML 开关和调参，不绑定代码
  4. 渐进式基础设施：Milvus 和 ES 都是可选的，搭了就用，没搭就跳过，最小可用只需要 MySQL
  5. 工程化不妥协：鉴权、敏感词、流式响应、离线评估——这些”非功能需求”才是企业级和 Demo 的分水岭

完整代码在 GitHub，clone 下来配好 MySQL 就能跑。
